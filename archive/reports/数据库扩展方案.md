# 数据库扩展方案

> 调研日期：2026年6月5日 | 基于2024年江苏省真实录取数据结构

---

## 一、核心发现：真实数据结构 vs 当前模拟结构

### 当前模拟数据的简化假设

```
admission_scores（当前）:
  每个大学 × 30个专业 → 直接生成录取分
  1个大学 = 1条"院校级"录取线（major_id=NULL 时）
  或 30条"专业级"录取线（major_id=具体专业时）
  所有专业共享同一选科假设（忽略选科要求）
```

### 真实数据的实际结构

```
江苏省投档线（真实）:
  每个大学 → N个"院校专业组" → 每个组内 M个专业
  专业组 = 选科要求 + 专业集合
  例: 南京大学2024年物理类有4个专业组
    07组(不限): 661分（包含经管/法学/外语等）
    08组(化学): 667分（包含计算机/电子/化学等）
    09组(化学): 663分（包含临床/药学等）
    11组(化学+中外): 647分（中外合作办学）
```

**结论：当前 `admission_scores` 表无法直接承载真实数据，需要扩展。**

---

## 二、扩展方案：渐进式三阶段

### 阶段 1：最小可行扩展（新增 2 字段）— 立即可做

在 `admission_scores` 表增加 2 个字段：

```sql
ALTER TABLE admission_scores
  ADD COLUMN subject_requirement VARCHAR(50) COMMENT '选科要求：不限/化学/化学+生物/...',
  ADD COLUMN major_group VARCHAR(50) COMMENT '专业组代号：07专业组/08专业组/...';
```

**优点**：
- 改动最小，不影响现有代码
- 可承载"院校专业组"级别的真实投档线
- 现有 `major_id` 用于专业级数据，新增字段用于专业组级数据

**缺点**：
- 专业组信息冗余存储（同一组内每条记录重复存选科要求和组名）
- 无法表达"哪些专业属于哪个专业组"的归属关系

**适用场景**：从 EOL/考试院获取投档线数据后直接导入，快速上线真实数据。

---

### 阶段 2：规范化扩展（新增 1 表）— 推荐方案

新增 `admission_groups` 表，将专业组作为独立实体：

```sql
-- 院校专业组表
CREATE TABLE admission_groups (
  id INTEGER PRIMARY KEY AUTO_INCREMENT,
  university_id INTEGER NOT NULL COMMENT '所属院校',
  group_name VARCHAR(50) NOT NULL COMMENT '专业组名称：07专业组',
  subject_requirement VARCHAR(100) NOT NULL COMMENT '选科要求：化学',
  year INTEGER NOT NULL COMMENT '招生年份',
  batch VARCHAR(30) DEFAULT '本科批' COMMENT '批次',
  selection_ratio VARCHAR(20) COMMENT '投档比例：1:1.05',
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  UNIQUE KEY uq_group (university_id, group_name, year),
  FOREIGN KEY (university_id) REFERENCES universities(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 扩展 admission_scores
ALTER TABLE admission_scores
  ADD COLUMN group_id INTEGER COMMENT '所属专业组',
  ADD FOREIGN KEY (group_id) REFERENCES admission_groups(id);
```

**新 ER 关系**：

```
universities (1) ──── (N) admission_groups ──── (N) admission_scores
                                                    │
majors (1) ────────────────── (N) ──────────────────┘
```

**优点**：
- 消除冗余，一个专业组的选科要求只存一次
- 支持"专业组→专业"的归属查询
- 可独立管理专业组信息（投档比例、招生简章链接等）

**缺点**：
- 需要修改 `generate_plan` SQL（增加 JOIN admission_groups）
- 数据导入需两步：先建组，再导入分数线

**适用场景**：正式生产环境，承载完整真实数据。

---

### 阶段 3：完整生产级（新增 3 表）— 未来架构

新增专业组归属、选科详情、招生计划三张表：

```sql
-- 专业组-专业归属表（哪些专业在哪个专业组内）
CREATE TABLE group_majors (
  id INTEGER PRIMARY KEY AUTO_INCREMENT,
  group_id INTEGER NOT NULL,
  major_id INTEGER NOT NULL,
  UNIQUE KEY uq_group_major (group_id, major_id),
  FOREIGN KEY (group_id) REFERENCES admission_groups(id),
  FOREIGN KEY (major_id) REFERENCES majors(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 选科要求详情表（支持"化学+生物"等多选）
CREATE TABLE subject_requirements (
  id INTEGER PRIMARY KEY AUTO_INCREMENT,
  group_id INTEGER NOT NULL,
  subject VARCHAR(20) NOT NULL COMMENT '科目：化学/生物/物理/不限',
  is_required TINYINT DEFAULT 1 COMMENT '1=必选 0=任选其一的',
  FOREIGN KEY (group_id) REFERENCES admission_groups(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 扩展 admission_scores 增加更多真实字段
ALTER TABLE admission_scores
  ADD COLUMN max_score INTEGER COMMENT '最高录取分',
  ADD COLUMN tuition INTEGER COMMENT '学费(元/年)',
  ADD COLUMN duration TINYINT COMMENT '学制(年)',
  ADD COLUMN campus VARCHAR(100) COMMENT '校区',
  ADD COLUMN enrollment_plan INTEGER COMMENT '招生计划数(专业级)',
  ADD COLUMN selection_line INTEGER COMMENT '投档线(专业组总分)';
```

**完整 ER 图**：

```
provinces
  │
universities ──── admission_groups ──── subject_requirements
  │                    │
  │                    ├── group_majors ──── majors
  │                    │                        │
  │                    │                        ├── major_profiles
  │                    │                        └── major_trends
  │                    │
  └── admission_scores ──────────────────────────┘
       (university_id, group_id, major_id, ...)
```

**适用场景**：完整商业产品，支持精确的"选科→可选专业组→可选专业"推荐链路。
